# 01 — Audit: current data vs v2 edoxen schema

## Audit findings

### Meeting YAMLs (`meetings/*.yaml`, 58 files)

Fields that **conflict** with the v2 Meeting schema:

| Field | Current | V2 schema | Fix |
|---|---|---|---|
| `year` | present | absent (derive from date_range) | DELETE |
| `virtual` | boolean | absent (derive from venue kind) | DELETE |
| `resolution_refs` | `["urn:..."]` | absent — should be `decisions: StructuredIdentifier[]` | RENAME + reshape |
| `source_urls[].kind` | `resolutions_pdf` | `enum: [agenda_pdf, minutes_pdf, decisions_pdf, report_pdf, register_url, landing_page]` | already valid |
| `city` | UN/LOCODE (e.g. `FRPAR`) | `pattern: "^[A-Z]{2}[A-Z0-9]{3}$"` | already valid |
| `country_code` | ISO alpha-2 | `pattern: "^[A-Z]{2}$"` | already valid |
| `type` | `plenary` / `conference_session` | `enum: [..., conference, ...]` | `conference_session` → `conference` |
| `status` | `completed` | `enum: [upcoming, completed, cancelled]` | already valid |
| Missing `visibility` | absent | `enum: [public, private, confidential]` | ADD `visibility: public` |

Fields that **match** correctly: `identifier`, `urn`, `ordinal`, `type`, `status`, `date_range`, `committee`, `general_area`, `city`, `country_code`, `localizations`, `source_urls`, `minutes`.

### Resolution YAMLs (`resolutions/*.yaml`, 56 files)

Fields that **conflict** with the v2 DecisionCollection schema:

| Field | Current | V2 schema | Fix |
|---|---|---|---|
| `resolutions` key | `resolutions:` | `decisions:` | RENAME |
| `identifier` (per-res) | string `"CIML/2009/1"` | `StructuredIdentifier[]` `[{prefix: "CIML", number: "2009/1"}]` | RESHAPE |
| `dates[].start` | `"2009-10-27"` | `dates[].date` | RENAME key |
| `dates[].kind` | `"decision"` / `"effective"` | `dates[].type` + enum check | RENAME key |
| `metadata.title` | string | `metadata.title` (legacy) or `metadata.title_localized[]` | already have title_localized in merged |
| `metadata.language` | `"en"` / `"fr"` | absent in v2 (language is per-localization) | DELETE |

Fields that **match** correctly: `doi`, `urn`, `agenda_item`, `localizations[].title`, `localizations[].subject`, `localizations[].actions[]`, `localizations[].considerations[]`, `localizations[].approvals[]`.

### Edoxen-data meetings (`edoxen-data/meetings/*.yaml`, 58 files)

These were generated by `convert_to_edoxen_model.rb` with **embedded** `decisions[]` containing full Decision objects. The v2 schema expects `decisions: StructuredIdentifier[]` (references only). Full Decision objects live in separate DecisionCollection files.

**Fix**: Either:
1. Keep the edoxen-data/ meetings with decisions as refs, and use resolutions/*.yaml as separate DecisionCollection files; OR
2. Update the v2 schema to allow embedded decisions (unlikely — the model intentionally separates them).

Going with option 1: the `edoxen-data/meetings/` files are the v2-compliant Meeting YAMLs with decisions as refs; `resolutions/*.yaml` are the v2-compliant DecisionCollection YAMLs.
