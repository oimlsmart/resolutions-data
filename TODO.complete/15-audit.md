# 15 — Architectural audit (2026-06-30)

A top-to-bottom pass over the OIML Resolutions browser. Findings are
grouped by layer; each finding notes the cost/risk of fixing it. The
intent is to surface structural debt before adding more features, not
to nitpick.

## Layer map

```
Edoxen schema (~/src/mn/edoxen/schema/edoxen.yaml)
        ↓ validates
Resolution YAMLs (resolutions/*.yaml, one per meeting)
        ↑ emitted by
Parser (scripts/author_yaml.rb ← reference-docs/.ocr/md/*.md)
        ↑ generated by
OCR pipeline (scripts/ocr/ — GLM-OCR API)
        ↑ sourced from
Manifest (scripts/manifest.yaml — 51 source PDFs)

Browser:
  Browser types (browser/src/types/resolution.ts)
        ↓ mirrors
  JSON mirror (browser/public/data/*.json)
        ↑ built by
  Build script (browser/scripts/build-data.mjs ← lib/transforms.mjs)
        ↓ consumed by
  Composables (browser/src/composables/*)
        ↓ rendered by
  Views (browser/src/views/*.vue) + App shell
```

## A. Data model

### A1. Type duplication between Meeting and Resolution
Both interfaces carry `city`, `country_code`, `venue`, `meeting_date`.
Resolution should REFERENCE its parent meeting (via `source_file`)
and inherit these — not duplicate them. **Cost:** low. **Risk:** low.
Fix: drop the duplicated fields on Resolution; resolve via
`getMeeting(resolution.source_file)`.

### A2. Magic strings for body type
`MeetingBodyType = 'ciml' | 'conference'` is hard-coded in TypeScript.
The list of body types is owned by `meeting-types.yaml`; the TS union
should derive from it (`type BodyType = keyof typeof meetingTypes`).
**Cost:** low. **Risk:** low.

### A3. Free-form `type: string` on Action / Consideration / Approval
Schema has a 60-value `enum` for action types; TS type is `string`.
Typos and unknown verbs silently pass through. The browser should
either widen the enum or accept the registry as the source of truth.
**Cost:** medium. **Risk:** medium (touches parser output too).

### A4. `is_acclamation: boolean` is too narrow
The current field can only express two states (acclamation vs not).
OIML resolutions can be: plenary vote, acclamation, secret ballot,
MA ballot (7372). Model should be `adoption_kind: AdoptionKind`.
**Cost:** low. **Risk:** low (parser already emits the kind via
identifier prefix `-acclaim-`).

### A5. Language code mismatch between TS and YAML
TypeScript uses ISO 639-1 (`'en' | 'fr'`). YAML uses ISO 639-3
(`'eng' | 'fra'`). `toLang6391()` bridges them but the bridge is
ad-hoc. Either:
- Make TS use ISO 639-3 everywhere (consistent), or
- Make YAML use ISO 639-1 (matches HTML lang, Intl.DateTimeFormat).
**Cost:** low. **Risk:** medium (touches every consumer of `lang`).

### A6. No value-object types for DOI / URN / AgendaItemId
Each is a `string` with no invariants. Should be a branded type
(e.g. `type Doi = string & { __brand: 'Doi' }`) so callers can't
accidentally pass a URN where a DOI is expected.
**Cost:** low. **Risk:** low.

### A7. Localization model not exposed in browser
The YAML has `Resolution.localizations[]`; the browser still treats
Resolution as a single flat row keyed by `language`. The
single-language row shape is what the JSON mirror emits — fine —
but the browser should have a `LocalizedResolution` concept that
groups sibling rows. **Cost:** medium. **Risk:** medium.

## B. Parser pipeline (scripts/author_yaml.rb)

### B1. Action verb lists are a 200-line constant
ACTION_PREFIXES + FR_ACTION_PREFIXES + EXTRA_EN_ACTION_PREFIXES.
Adding a new verb means editing Ruby. The verbs should live in a
YAML registry (`scripts/action-prefixes.yaml`), and the parser
should look them up at runtime. **Cost:** low. **Risk:** low.

### B2. Subject detection is fragile
`extract_subject` matches `The Committee,` / `The Conference,` on a
single line. OCR occasionally wraps the subject across lines or
adds invisible whitespace. Result: some resolutions have no
subject. **Cost:** low. **Risk:** low.

### B3. No parser tests
The parser handles 3 OCR shapes (modern, older, decisions-narrative)
plus bilingual splits. No spec coverage. Adding a fixture-per-shape
test suite would prevent regressions. **Cost:** medium. **Risk:** low.

### B4. `parse_narrative` synthesizes titles when the section heading is empty
Some CIML 39–42 sections have a "## N." heading with no title. The
parser falls back to `synthesize_title` which takes the first 14
words of the body. This is unfaithful to the source. Should emit
`(Untitled)` and let the editor fix it manually. **Cost:** low.
**Risk:** low.

### B5. Pre-2012 resolutions have no agenda_item
The parser only extracts agenda_item when the source markdown has
"Agenda item N" line. CIML 39–42 narrative docs don't have this.
The model already allows `agenda_item` to be optional; nothing to
fix here beyond documenting the boundary.

## C. Build pipeline (browser/scripts/)

### C1. `buildResolutionRecord` does too much
It: validates the row, flattens the localization, normalizes the
snippet, computes the DOI, fills in legacy fallbacks. Split into
3-4 single-purpose functions: `flattenLocalization`,
`normalizeSnippet`, `computeResolutionId`, `pickSourceUrl`.
**Cost:** low. **Risk:** low.

### C2. Schema validation never runs in CI
The build script reads YAML with js-yaml; it doesn't validate against
the edoxen schema. A `npm run validate` script that runs the edoxen
Ruby CLI on every YAML would catch regressions.
**Cost:** low. **Risk:** low.

### C3. No incremental rebuild
Every `npm run build` re-parses all 28 YAMLs. With 1,515 resolutions
this is fast (<1s) but the architecture doesn't support caching.
Acceptable for now.

### C4. JSON mirror has no schema
The `public/data/*.json` files are typed only via TS interfaces. A
JSON Schema alongside would let external consumers validate.
**Cost:** medium. **Risk:** low.

### C5. ResolutionDetail chunk is 782KB
Most of it is `@asciidoctor/core`. Should be lazy-loaded only when
a resolution actually has rich-text actions. **Cost:** low.
**Risk:** low.

## D. Browser (Vue)

### D1. `useResolutions` is a singleton with module-level state
The composable stores `resolutions` and `isLoaded` at module scope.
This works because there's only ever one resolution set, but it
breaks SSR testability. Acceptable pattern for vite-ssg.

### D2. No URL redirects from legacy `/-en` / `/-fr` paths
After item 15, URLs changed from `/meetings/ciml-39-decisions-en`
to `/meetings/ciml-39-decisions`. Old links 404. Need a static
redirect map generated at build time. **Cost:** low. **Risk:** low.

### D3. `canonicalMeetingId` strips a suffix that no longer exists
The function lives in `useMeetings.ts` and strips `-en`/`-fr` from
source_file. After item 15, no source_file carries those suffixes.
The function is now a no-op identity. Should be removed.
**Cost:** low. **Risk:** low.

### D4. `Meeting.language: '' | 'en' | 'fr'` is wrong post-item-15
After the glossarist refactor, a meeting has multiple languages
exposed via `languages: string[]`. The single `language` field is
stale. Should be removed in favor of `languages[0]`.
**Cost:** low. **Risk:** low.

### D5. Duplicate rendering logic across views
Home.vue, MeetingDetail.vue, ResolutionDetail.vue each render
"action chips" with the same `getActionColor()` call. Should be a
shared `<ActionChips :actions=... />` component. **Cost:** medium.
**Risk:** low.

### D6. No image optimization
SVGs (logos, icons) are served raw. PNGs (favicons) aren't
responsive. For a documentation site, low priority.

### D7. `formatDateRange` has a bug in the same-month branch
The current implementation does string surgery on the Intl output
that breaks for non-English locales. Should be: format day range
manually, then append the localized month/year.
**Cost:** low. **Risk:** low.

## E. i18n

### E1. Action vocabulary NOT localized
`formatActionType('thanks')` returns `'thanks'`. Should return
`'Remerciments'` in FR. Add `labels: { eng, fra }` to each entry in
`action-types.yaml`; expose `getActionLabel(type, lang)`.
**Cost:** low. **Risk:** low.

### E2. Date formatter dispatches by ISO 639-1
`Intl.DateTimeFormat('en')` works for 2-letter codes. After item 15,
the canonical lang is ISO 639-3 (`'eng'`). The bridge `toLang6391()`
exists in `transforms.mjs` but not in `useDateFormat`. Should be
centralized. **Cost:** low. **Risk:** low.

### E3. No pluralization
`interpolate('{count} resolutions', { count: 1 })` → "1 resolutions".
Should be "1 resolution". Use Intl.PluralRules or a small
`pluralize()` helper. **Cost:** low. **Risk:** low.

### E4. No RTL support
If Arabic is added (not currently planned), the layout will break
because nothing uses logical CSS properties (`margin-inline-start`
vs `margin-left`). Acceptable for now; document the assumption.

## F. SEO

### F1. No per-page meta tags
Each page should set `<title>`, `<meta name="description">`, OG
tags, Twitter card. `@unhead/vue` is installed but unused.
**Cost:** medium. **Risk:** low.

### F2. No JSON-LD structured data
Each resolution page should emit `schema.org/Legislation` or
`schema.org/CreativeWork` JSON-LD. This is how Google renders rich
snippets. **Cost:** medium. **Risk:** low.

### F3. No canonical URLs
Search engines see both `/resolution/CIML-2025-44` and
`/resolution/CIML-2025-44?lang=fr` as separate pages. Need
`<link rel="canonical">` per page. **Cost:** low. **Risk:** low.

### F4. No hreflang alternates
For EN/FR variants, `<link rel="alternate" hreflang="...">` should
be emitted so search engines know which version to show.
**Cost:** low. **Risk:** low.

### F5. Sitemap lacks lastmod / changefreq
Current sitemap is just URLs. Adding `<lastmod>` from the YAML
`metadata.dates[0].start` would help crawler scheduling.
**Cost:** low. **Risk:** low.

## G. Performance

### G1. ResolutionDetail chunk is 782KB (gzip 212KB)
Mostly `@asciidoctor/core`. Lazy-load only when the resolution has
rich-text actions. **Cost:** low. **Risk:** low.

### G2. JSON mirror is 1+ MB uncompressed
`public/data/resolutions.json` ships as one file. Could be split
per-meeting for parallel fetch. Low priority — gzip brings it to
~300KB.

### G3. No preloading of prev/next resolution
The detail page has prev/next nav. Preloading the next resolution's
JSON record would make nav feel instant. **Cost:** low. **Risk:** low.

---

## H. Testing

### H1. No tests
Zero test files. Should add:
- Parser spec (Ruby RSpec) — at least one fixture per OCR shape.
- Transform spec (Node test) — `buildResolutionRecord` inputs/outputs.
- Composable spec (Vitest) — `useMeetings`, `useResolutions`,
  `useDateFormat`.
- YAML data spec — every meeting file validates against edoxen
  schema; every resolution has at least one localization.
**Cost:** high. **Risk:** low.

### H2. No CI lint
No ESLint config, no Prettier config. Adding both is cheap and
catches many bugs. **Cost:** low. **Risk:** low.

## Prioritization

| P0 | D2 (URL redirects) | Old links 404 — user-facing breakage |
| P0 | E1 (action vocab i18n) | Visibly broken for FR users |

| P1 | A1, A2, A4 (model cleanup) | Pays off every future feature |

| P3 | C4 (JSON Schema) | External consumers only |

## Things that are working well (don't break)

- Per-meeting YAML model with `localizations[]` (glossarist pattern)
- YAML + thin TS wrapper convention (cities, meetingTypes,
  action-types, committee, countries, country-flags, translations)
- `useDateFormat` composable
- SSG pre-rendering (vite-ssg)
- The build pipeline (build-data.mjs) is simple and fast
- The TypeScript types module (`types/resolution.ts`) is
  free-standing with no Vue/reactivity imports

## What to add next (architecturally)

- A **`browser/src/domain/`** module: typed value objects
  (`Doi`, `Urn`, `AgendaItemId`, `Iso639Code`, `Iso3166Code`,
  `IataCityCode`), enum registries (`AdoptionKind`, `ActionType`),
  and a `LocalizedText` branded type. Today these are strings
  scattered across types and code paths.
- A **`browser/src/components/`** directory: extract shared
  rendering pieces (`<ActionChips>`, `<MeetingLinkBadge>`,
  `<ResolutionRow>`, `<UrnCopyBar>`). Today these are inline in
  views, duplicated 2-3 times each.
- A **`scripts/specs/`** directory: RSpec or Minitest fixtures for
  the parser; one fixture per OCR shape (modern, older, decisions).
- A **`browser/scripts/validate.mjs`** that runs the edoxen schema
  validator on every YAML; wires into `npm run build`.
