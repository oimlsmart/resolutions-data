# 01 — Meeting validation in CI

## Problem

The meeting YAMLs in `meetings/` are generated by `scripts/generate_meetings.rb`
but nothing checks that they conform to the canonical Edoxen Meeting schema.
The deployed site can silently ship broken meeting metadata.

Two concrete issues surface when running `edoxen validate-meetings meetings/*.yaml`:

1. **Virtual meetings emit `city: null` / `country_code: ''`** — the schema
   requires `city` to be a 5-char UN/LOCODE string and `country_code` to be
   a 2-letter ISO 3166-1 string. Both are optional, but if present must
   match the pattern. The generator unconditionally includes the keys even
   when the value is nil/empty, which fails validation.

2. **No CI step runs `edoxen validate-meetings`** — `.github/workflows/deploy-pages.yml`
   only runs `ruby scripts/validate_yaml.rb` (which checks `resolutions/`).

## Plan

1. Update `scripts/generate_meetings.rb` to omit `city` and `country_code`
   from the emitted YAML when the value is nil or empty. Virtual meetings
   should still set `virtual: true` so the metadata is unambiguous.
2. Re-run the generator to refresh `meetings/*.yaml` in place.
3. Verify with `edoxen validate-meetings meetings/*.yaml` — should be
   100% valid.
4. Add a `validate-meetings` step to `.github/workflows/deploy-pages.yml`
   that installs the `edoxen` gem and runs the validator over `meetings/*.yaml`.

## Acceptance

- [x] `edoxen validate-meetings meetings/*.yaml` reports 0 failures locally
- [x] CI runs `edoxen validate-meetings` on every PR
- [x] PRs that break meeting YAMLs are blocked from merge
